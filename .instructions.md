## How This Works (For AI Tool Context)

### The Identity Flow
```
1. User login via PingFederate (OIDC)
2. Portal stores token in session
3. Portal sends Bearer token to agent via PingAccess
4. PingAccess validates token + checks authorization (PingAuthorize)
5. Portal exchanges token for agent-scoped token (RFC 8693)
6. Agent receives scoped token (limited to that agent only)
7. Agent invokes MCP tools with scoped token
8. MCP tool validates token + enforces data-level access control
```

### Key Innovation: RFC 8693 Token Exchange
- **User token** (broad privileges) → **Agent token** (narrow scope)
- Each agent gets unique audience claim (aud)
- One hacked agent can't access other agents' scope
- **Defense in depth** at token layer

## For AI Tools: Common Tasks

### Deploy to k3d
1. Check `.env` has credentials: `PING_IDENTITY_DEVOPS_USER`, `PING_IDENTITY_DEVOPS_KEY`
2. Run: `make setup`
3. Wait for rollout: `make status`
4. Access: https://portal-external.localhost/workforce-portal

### Add New Agent
1. Create new directory: `cp -r acp-trip-planner-agent acp-my-agent`
2. Update `src/router.py`: Change agent identity (aud claim)
3. Register in Kubernetes: Add deployment manifest
4. Update portal config: Add endpoint to `acp-workforce-portal/js/config.js`
5. Rebuild + redeploy: `docker build ... && kubectl rollout restart deployment/acp-my-agent`

### Fix Common Issues
- **404 errors**: Check portal config.js endpoint paths
- **"Registry empty"**: Already handled by LLM fallback (in router.py)
- **Token validation fails**: Verify scoped token has correct aud claim
- **MCP tool 403**: Check user authorization policy in PingAuthorize
